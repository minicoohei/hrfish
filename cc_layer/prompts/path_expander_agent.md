# PathExpanderAgent プロンプトテンプレート

## モデル: sonnet（パスごとに1つ、5並列起動）

## Skills 参照
- mirofish-zep: Zepグラフからの背景情報取得
- mirofish-career: イベントタイプ定義・ブロッカールール・スコアリング重み

## タスク

1つのキャリアパスについて、**10年間を4期に分け、さらに確率に基づいて4シナリオに分岐**させて展開してください。

パスの方向性（例: 「LayerXで経営層へ」）は共通だが、以下の4シナリオを場合分けして描写します:
- **best**: 最良ケース — すべてがうまくいった場合
- **likely**: 標準ケース — 最も起こりやすい着地点
- **base**: ベースケース — 現状維持に近い、特段の成長がない場合
- **worst**: ワーストケース — 転職失敗、リストラ、健康問題、市場崩壊など**本当のダウンサイド**。「やや成長が鈍い」程度はworstではない

## 出力JSON

各パスは **scenarios（4個: best/likely/base/worst）** を持つ。全シナリオの Period 1-2 は共通（分岐前）、
Period 3-4 でシナリオが分岐する構造。

**分岐の考え方:**
- Period 1-2（Year 1-4）: 全シナリオ共通。この期間の努力・成果は確定的
- Period 3（Year 5-7）: **分岐点**。ここでの昇進/調達/転職の成否で分かれる
- Period 4（Year 8-10）: 各シナリオごとに異なる着地点

**⚠️ 4シナリオの年収レンジ設計ルール:**
- **年収の単位は「万円」。** 円で書かないこと（例: 800 = 800万円 ✅, 8000000 ❌）
- **best と worst の年収差は最低2倍以上** を目安にすること
- **worst の年収は現在年収を下回ることもありうる。** 転職失敗・起業頓挫・リストラ・健康問題・介護離職等の現実的リスクを反映すること
- **base は「現状維持」** — 大きなリスクも取らないが成長もしない場合。年収は現在の±20%程度
- worst の satisfaction は best より **少なくとも 0.3 ポイント以上** 低くすること

❌ **NGパターン:**
- best=2500万, worst=2200万（差が小さすぎ）
- best=2500万, base=2400万, worst=2300万（全部同じ。シナリオ分析の意味がない）
- base が worst より低い（base は中間、worst が最悪であるべき）
- worst が「成長が鈍い」程度（それはbase。worstは失職・減収・健康問題レベル）

✅ **OKパターン:**
- best=2500万, likely=1500万, base=1000万, worst=500万
- best=3000万, likely=1800万, base=900万, worst=400万（起業失敗+転職市場冷え込み）

**worst シナリオ必須チェックリスト（1つ以上該当すること）:**
- [ ] 年収が現在年収を**下回る**
- [ ] 転職失敗・リストラ・会社倒産・事業撤退
- [ ] 健康問題（バーンアウト、休職、適応障害）
- [ ] 家庭事情（介護離職、離婚による生活変化）
- [ ] 市場崩壊（業界縮小、技術陳腐化、不況）
- [ ] 起業失敗（資金ショート、顧客獲得不振）

**worst の satisfaction は 0.5 以下** であること。worst で satisfaction 0.7 はありえない。

**worst satisfaction の下限はパスの深刻度に応じて差別化すること:**
- 経済的破綻（廃業・失業6ヶ月超）: satisfaction 0.15-0.25
- キャリア停滞（昇進見送り・年収微減）: satisfaction 0.30-0.40
- 精神的消耗（バーンアウト・適応障害）: satisfaction 0.10-0.20
全パスのworst satisfactionが0.35-0.45の狭いバンドに集中するのはNG。パス間で最低0.10の差をつけること。

**satisfaction の最低スプレッド:**
- best と worst の satisfaction 差は **0.3以上** であること
- 5パス間の satisfaction 差（最大 - 最小）は **0.15以上** であること
- 全パスが satisfaction 0.7-0.8 に収束するのはNG

**確率配分の目安:**
- best: 10-20%、likely: 35-45%、base: 25-35%、worst: 10-20%
- 4つの合計が100%になること

**シナリオ確率の注意:**
- best/likely/base/worst の確率は、パスの内容に応じて個別に設定すること
- 以下のテンプレート的な確率セットは使用禁止:
  NG: {0.15, 0.40, 0.30, 0.15}, {0.12, 0.40, 0.30, 0.18}, {0.12, 0.38, 0.30, 0.20}
- 確率は0.01刻みの連続値で設定し、probability_rationaleに具体的根拠を記載すること
- 例: best=0.07（業界のIPO到達率5-8%にスキル補正）, likely=0.43（転職成功率60%×定着率70%）

**likely と base の年収差の最低保証:**
- likely と base の final_state.annual_income の差は最低200万円以上とすること
- 差が200万未満の場合は、baseシナリオの内容をlikelyと質的に異なるものに再設計すること

## 展開ガイドライン

1. **各期は独立した物語**: 前期の結果が次期の前提条件になる因果関係を明示
2. **snapshot は全 period で必須**: common_periods・各 scenario の periods すべてに snapshot を付与すること。省略は不可。年収(万円)、満足度(0-1)、ストレス(0-1)、WLB(0-1)の4フィールドすべてが必要。

⚠️ **以下はNG（バリデーションエラーになる）:**
```json
// ❌ snapshot がない period
{"period_name": "Year 5-7", "narrative": "..."}

// ❌ snapshot のフィールドが欠けている
{"period_name": "Year 5-7", "snapshot": {"annual_income": 1200}}
```

✅ **正しい period:**
```json
{"period_name": "Year 5-7", "narrative": "...",
 "snapshot": {"annual_income": 1200, "satisfaction": 0.75, "stress": 0.45, "work_life_balance": 0.6},
 "events": [...]}
```

3. **events**: mirofish-career の24イベントタイプから選択（promotion, job_change, skill_upgrade, health_issue, family_event 等）
4. **blockers_active**: 各期でアクティブなブロッカーをリスト（mortgage_pressure, childcare_time, health_risk, spouse_career, aging_parent, relocation_constraint）
5. **key_decision**: その期で候補者が下した最も重要な判断を1文で
6. **final_state.total_score**: 0のまま（Python CLIが後で計算）

**final_state は必須。以下のフィールド名を厳密に使用すること: annual_income, satisfaction, stress, work_life_balance**

### 報酬水準リファレンス（最重要）

**年収を設定する際は、必ず以下の水準を参照すること。** SubAgentが「日本の一般的なサラリーマン」の感覚で数字を出すと、経営幹部・金融・外資系の報酬を大幅に過小評価する。

**テーブルの読み方:**
- **P10**: 下位10%ライン。この水準を下回るのは全体の10%。**worst/baseシナリオの下限**
- **P50**: 中央値。そのポジションの人の半数がこの水準以上。**likelyシナリオの基準**
- **P90**: 上位10%ライン。この水準を超えるのは全体の10%。**bestシナリオの上限**
- **到達率**: 同期入社者（またはそのキャリアパスの母集団）のうち、そのポジションに到達する割合（%）。**組織はピラミッド構造であり、上位ポジションほど到達率は急激に下がる**
- **異常値帯**: P90を超える例外的なケース。**シミュレーションでは原則使わない**（参考情報のみ）

**組織ピラミッドの基本構造（大企業の同期入社者ベース）:**
```
                     取締役 (0.1%)
                   執行役員 (0.5%)
                  部長 (4-10%)
                課長 (30-50%)
              係長・主任 (50-70%)
            一般社員 (残り全員)
```
- Sources: 厚労省「賃金構造基本統計調査」部長3.8%/課長7.1%、パーソル総合研究所(2022)、東洋経済/ダイヤモンド人事部長調査

**シナリオ別の年収設定ルール:**
- **worst**: P10以下（降格、失職、減収。P10の70%まで下がるケースも）
- **base**: P10〜P50の間
- **likely**: P50付近（±10%）
- **best**: P50〜P90の間
- **異常値帯の値はbestでも使わない**（P90が上限）

**到達率とprobability_noteの連携:**
- 到達率5%未満のポジションへのキャリアパスでは、probability_noteで到達確率の低さに必ず言及すること
- 例: 「メガバンク取締役（到達率0.1%）への昇進を前提としたパス。第一選抜に残り続ける必要があり...」
- 到達率が低いほど、そのポジションに到達するbestシナリオの全体probability自体も低くなるべき

### probability_note の書き方

**「なぜその確率か」を候補者固有の文脈で説明する。**

悪い例: `"probability_note": "昇進は難しい"`
良い例: `"probability_note": "CPOは1社に1席で、LayerXでは福島CEOがプロダクト権限を握っている。PdM歴9年の実績は強いが、創業メンバーが優先されるため、VP→CPO昇進は10%程度。"`

### パス全体の確率

パスのトップレベルにも `overall_probability` と `probability_rationale` を付与する。
これは各イベントの複合確率ではなく、「このパスの最終状態（final_state）に10年以内に到達する全体的な確率」を示す。
